Claude iOS Troubleshooting

Claude iOS App 卡旧账号 / Something went wrong：用 Charles 清理旧 session 的完整攻略

一次真实排障记录：从社区 Proxyman 思路出发，最后用 Charles 精确拦截 /api/account*，让 Claude iOS App 自己清掉本机旧登录态。

发布于 2026-06-14 · 适合新手按步骤操作

这篇不是“解封教程”，也不是绕过 Claude 的账号风控。它解决的是 iOS App 本机旧登录态卡住的问题。如果账号本身已经被服务端限制，这个方法不会恢复账号权限。

最终有效方案一句话

在 Mac 上用 Charles 做 iPhone 的 HTTP/HTTPS 代理，只拦截 Claude iOS App 访问 https://claude.ai:443/api/account* 的请求，把响应改成 401 session_expired，并返回两个过期的 Set-Cookie。App 收到“会话过期”后，会主动清理本机旧 session，再关闭代理重新打开 Claude，就可以重新登录。

适用和不适用场景

适合

  • Claude iOS App 打开就报 Something went wrong。
  • App 像是一直记住旧账号，删除、重装仍然不干净。
  • 手机其他网络正常，但 Claude App 自己卡住。
  • 你只是想清理本机旧登录态，重新登录自己的可用账号。

不适合

  • 账号在服务端确实被限制、封禁或无法使用。
  • Mac 和 iPhone 不在同一个 Wi-Fi。
  • 你不愿意临时信任 Charles 证书。
  • 你不准备在修好后关闭代理和证书信任。

背景：为什么一开始试了 Proxyman

社区最早的思路是用 Proxyman 拦截 Claude App 请求，在请求里删除旧的 sessionKey 和 routingHint Cookie，再在响应里通过 Set-Cookie 让 iOS 删除这两个旧值。

这个思路是对的：核心不是“伪造登录”，而是让 App 不再带着旧 session 继续撞墙。但在这次实际操作里，Proxyman 对部分 Claude/Anthropic 域名开启 SSL Proxying 后，App 容易显示 You are offline。最后换成 Charles，并且只拦截更窄的路径：claude.ai:443 下的 /api/account*。

准备工作

  • 一台 Mac。
  • 一台 iPhone。
  • Mac 和 iPhone 连同一个 Wi-Fi。
  • Mac 上安装 Charles。
  • 知道 Mac 的局域网 IP，例如 192.168.1.253。

查 Mac IP 可以用：

ipconfig getifaddr en0

Charles 默认 HTTP 代理端口通常是 8888。如果你改过端口，以 Charles 里显示的为准。

VPN 建议：排错阶段先关闭 iPhone 上的 VPN。如果 Claude 需要代理出网，优先让 Mac 开 VPN，iPhone 只通过 Wi-Fi HTTP 代理连到 Mac 的 Charles。iPhone 同时开 VPN 时，很多 VPN App 会接管路由，导致 Charles 看不到请求。

第一步：让 iPhone 走 Charles 代理

  1. 打开 iPhone 的 设置，进入当前 Wi-Fi。
  2. 点 Wi-Fi 右侧的信息按钮，找到 配置代理。
  3. 选择 手动。
  4. 服务器填 Mac IP，例如 192.168.1.253。
  5. 端口填 8888，然后保存。
  6. 先用 Safari 打开 http://example.com 测试普通网页是否有网。

如果这一步显示“无互联网连接”，优先检查 Charles 是否允许 iPhone 访问：看到弹窗就点 Allow，或者在 Charles 的 Access Control 里允许 iPhone IP。

第二步：安装并信任 Charles 证书

在 iPhone 已经连上 Charles 代理的前提下，用 Safari 打开：

http://chls.pro/ssl
  1. 设置 -> 通用 -> VPN 与设备管理，安装 Charles 证书描述文件。
  2. 设置 -> 通用 -> 关于本机 -> 证书信任设置，打开 Charles 根证书的完全信任。

这一点非常关键。我们这次中间有一段时间一直看到 You are offline，本质上就是证书没有被完整信任，HTTPS 解密失败。

第三步：只开启 claude.ai 的 SSL Proxying

在 Charles 里打开：

Proxy -> SSL Proxying Settings

勾选 Enable SSL Proxying，然后只添加这一条：

Host: claude.ai
Port: 443

不要一开始就把 *.anthropic.com、a-api.anthropic.com、*.claude.ai 全部加进去。拦截范围太大时，Claude App 更容易把自己判成离线。

第四步：新增 Rewrite 规则

打开 Tools -> Rewrite，新建一个 Set，例如 Claude iOS clear stale session，并勾选 Enable Rewrite。

Location

Protocol: https
Host: claude.ai
Port: 443
Path: /api/account*

Rule 1：把响应状态改成 401

401 Unauthorized

Rule 2：把响应 body 改成会话过期

{"type":"error","error":{"type":"session_expired","message":"Session expired"}}

Rule 3：让 sessionKey 过期

Header: Set-Cookie
Value: sessionKey=; Path=/; Domain=.claude.ai; Expires=Thu, 01 Jan 1970 00:00:00 GMT; Secure; HttpOnly; SameSite=Lax

Rule 4：让 routingHint 过期

Header: Set-Cookie
Value: routingHint=; Path=/; Domain=.claude.ai; Expires=Thu, 01 Jan 1970 00:00:00 GMT; Secure; HttpOnly; SameSite=Lax

重点：这两个 Set-Cookie 必须是响应头，不是请求头。

第五步：正式执行

  1. iPhone 上彻底关闭 Claude App。
  2. 确认 iPhone Wi-Fi 代理仍然指向 Mac 的 Charles。
  3. 确认 Charles 开着 SSL Proxying 和 Rewrite。
  4. 打开 Claude App，让它请求一次账号接口。
  5. 如果 Charles 里看到 https://claude.ai/api/account... 被改成 401，就说明命中了。
  6. 再次彻底关闭 Claude App。
  7. 回到 iPhone Wi-Fi 设置，把代理改回关闭。
  8. 重新打开 Claude App。
  9. 正常情况下，旧登录态已经清掉，可以重新登录。

成功后不要继续开着 Rewrite，否则你每次访问账号接口都会被 Charles 改成 session expired。

常见问题排查

现象最可能原因处理方式
iPhone 设置代理后无互联网连接Charles 没允许 iPhone 访问，或 IP/端口填错确认 Mac IP 和端口，Charles 里点 Allow，或允许局域网网段
Safari 打开 HTTPS 页面报证书错误Charles 证书只安装了，没有完全信任重新走 chls.pro/ssl，安装描述文件，再打开根证书信任
Claude App 显示 You are offlineHTTPS 解密失败，或 SSL Proxying 范围太大只保留 claude.ai:443，关掉其他 SSL Proxying
仍然 Something went wrongRewrite 没命中 /api/account*检查 Location 的 Host、Port、Path；确认 Charles 能看到 HTTPS 明文请求
Charles 里只有 CONNECT，看不到路径证书没信任，或 SSL Proxying 没开对 Host重做证书信任，只给 claude.ai:443 开 SSL Proxying
开手机 VPN 后 Charles 没流量VPN 接管了网络路径先关 iPhone VPN；需要外网时让 Mac 负责出网

收尾清理

  1. iPhone Wi-Fi 代理改回关闭。
  2. Charles 里关闭 Rewrite。
  3. Charles 里关闭 claude.ai 的 SSL Proxying，或者直接退出 Charles。
  4. 如果短期不再抓包，可以在 iPhone 里删除 Charles 描述文件，并关闭根证书信任。

HTTPS 抓包工具很强，也很危险。只在自己设备、自己账号、自己明确知道目的的情况下使用。

这次真正有用的判断

这次排查里，最容易误判的是把所有 Claude/Anthropic 域名都加进 SSL Proxying。看起来“抓得更全”，但实际会让 App 更容易离线。最后能成功，是因为把范围缩到最小：

只拦截 claude.ai:443
只改 /api/account*
只让 App 收到 session_expired
命中一次后立刻关代理

这也是给其他类似 App 问题的通用经验：不要一开始就全量拦截，先找最小的状态刷新接口，让客户端自己走退出登录或会话过期逻辑。
